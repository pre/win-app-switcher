win-app-switcher — Rewrite Plan

macOS-style application switching for Windows, written in Rust. Reference implementation: AltAppSwitcher (C, studied at ~/dev/rnd/AltAppSwitcher).

1. Goal

A single small background process that makes the Windows key behave like the macOS Command key for application switching. Target audience is the power user: speed, reliability and predictability over feature count.

2. Scope — decisions

Kept (from the spec and AltAppSwitcher review)

  • WIN + TAB — macOS-style app switcher dialog: one icon per running application, horizontal row.
  • WIN + SHIFT + TAB — move selection backwards.
  • WIN + § — window switcher for the active application. Quick tap = instant switch to the next window, no dialog. WIN held = vertical list dialog, one row per window: app name | app icon | window title. § advances, SHIFT + § goes back.
  • Mouse hover selects — hovered icon shows the selection border; releasing WIN activates it. Left click activates immediately; click outside cancels.
  • Arrow keys move the selection (Left/Right in app switcher, Up/Down in window list). No hjkl.
  • ESC cancels the dialog without switching.
  • WIN + Q — close all windows of the selected app (app switcher only). No hover “X” close button.
  • Theme — auto (follow Windows light/dark) / light / dark.
  • Scale — dialog size multiplier from config.
  • Selected app name shown under the selected icon only; other icons bare.
  • Dialog placement — monitor under the mouse cursor, or primary (config).
  • Virtual desktop filter — current desktop only, or all (config).
  • Restore minimized windows on activation (config).
  • Elevated apps — applications running as administrator (e.g. VS Code launched from WSL as superuser) are listed and switchable like any other app. Requires the switcher itself to run elevated (see §4).
  • Single instance — second launch shows a Yes/No dialog: “already running — restart it?”.
  • Tray icon — “§” glyph; right-click menu with Quit.
  • TOML config + fully commented-out config.example.toml.

Dropped (deliberately)

  • Updater.exe and Settings.exe — config file only.
  • Remappable keybindings — WIN+TAB / WIN+§ / SHIFT / Q / arrows / ESC are hardcoded. Small config stays small.
  • Hover “X” close button, hjkl navigation.
  • App-name labels for all/none (AAS display name modes) — always “selected only”.
  • App filter by mouse monitor (AAS app filter mode) — always list all eligible windows.
  • Gnome/window-per-entry switcher mode (AAS app switcher mode: window).
  • Mouse/keyboard common selection option — mouse and keyboard selections are independent, as in macOS.
  • Update check, .bat helper scripts — startup via a documented shortcut in shell:startup (README section).
Key insight from the reference code: AltAppSwitcher’s “window mode” (alt+~) shows no UI at all — it live-switches windows on each key press. Our WIN+§ list dialog is therefore a new design, not a port. The quick-tap behavior, however, matches what AAS already proves works.

3. UX specification

App switcher (WIN + TAB)

  • On WIN+TAB the dialog opens immediately, selection starts at the second app (index 1 — pressing and releasing instantly switches to the previous app, macOS behavior).
  • Apps ordered by z-order of their topmost window (most recently used first).
  • Repeated TAB while WIN held advances; SHIFT+TAB / Left / Right move the selection; wrap-around at the ends.
  • Releasing WIN activates the selection (keyboard selection, or hovered icon if the mouse moved last).
  • Rendering: rounded-corner panel, one icon per app group, selection shown as a rounded highlight behind the icon, selected app’s name centered underneath.

Window switcher (WIN + §)

  • On § down (WIN held): the next window of the foreground app is selected immediately. If WIN is released before dialog_delay_ms elapses, the switch happens with no UI.
  • If WIN is still held after the delay, the list dialog appears: rows of app name | icon | window title, current selection highlighted.
  • § moves down, SHIFT+§ moves up, Up/Down arrows likewise; wrap-around; ESC cancels; WIN release activates.
  • Windows ordered by z-order within the app group.

The § key is identified by scan code 41 (the key left of digit 1: § on Finnish/Swedish layouts, ` on US, ² on French…). AltAppSwitcher does the same via its USKeyToLocalKey translation. All other keys use virtual-key codes, so the shortcuts work on any layout.

4. Architecture

Elevation & priority

  • The switcher runs elevated (as administrator). This is a hard requirement, not a preference: UIPI blocks a non-elevated process’s low-level keyboard hook from receiving any input while an elevated window has focus — WIN+TAB would simply not fire inside an elevated VS Code. Elevation is also needed to attach input to and activate elevated windows. Enable SeDebugPrivilege at startup (as AAS does).
  • REALTIME_PRIORITY_CLASS for the process, THREAD_PRIORITY_TIME_CRITICAL for the hook thread — switching must be reliably fast even under full CPU load (compiles, games). REALTIME requires elevation (SeIncreaseBasePriorityPrivilege); since we require elevation anyway, it is available. If launched unelevated regardless, Windows silently falls back to HIGH — degraded but functional for non-elevated apps. Safe at REALTIME because every thread is event-driven (blocked in GetMessage); no busy loops anywhere.

Process & threads

  • Hook thread: installs WH_KEYBOARD_LL, decides swallow/pass per event, posts Init/Next/Prev/Deinit messages to the main thread. The callback must stay fast (Windows silently drops slow hooks), so it does nothing but state transitions and PostThreadMessage.
  • Main thread: message loop owning the tray icon, config, and the dialog windows.

WIN key handling (the hard part)

  • Swallow WIN+TAB, WIN+§ and WIN+Q in the hook; pass every other WIN combo through so WIN+L, WIN+D etc. keep working.
  • Once a switcher session starts, the bare WIN release must not open the Start menu. Standard trick (AAS does a variant of this): inject a no-op key event (e.g. VK_NONAME or right-ctrl down/up) before the WIN up passes through, so Windows sees WIN as a combo, not a tap.
  • Ignore injected events (LLKHF_INJECTED) to avoid feedback loops.
  • On session end, verify no key is left logically stuck (AAS’s RestoreKey exists because this genuinely happens; we replicate the check with GetAsyncKeyState + corrective SendInput).

Window enumeration & grouping

  • EnumWindows + eligibility filter, ported from AAS IsEligibleWindow: visible; not WS_EX_TOOLWINDOW; not topmost-without-appwindow; not owned (unless WS_EX_APPWINDOW); class-name blocklist (Shell_TrayWnd, Windows.UI.Core.CoreWindow, …); not DWM-cloaked; optionally on the current virtual desktop (IVirtualDesktopManager::IsWindowOnCurrentVirtualDesktop).
  • Group windows by resolved process executable path — via OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION) + QueryFullProcessImageName, which works across integrity levels (elevated targets included). UWP special case: for ApplicationFrameWindow hosts, resolve the real child PID (port AAS FindActualPID).
  • Activation: AttachThreadInput to the foreground thread + SetForegroundWindow, restore from minimized per config (port AAS’s proven sequence, including the hung-foreground timeout guard in TryAttachToForeground).

Icons

  • Win32 apps: SHGetFileInfo / IShellItemImageFactory on the exe path → large icon.
  • UWP apps: IShellItemImageFactory on shell:AppsFolder\<AUMID> — one code path, replaces AltAppSwitcher’s ~300 lines of appx-manifest parsing.
  • Cache icons per exe path for the lifetime of the process.

Rendering

  • Direct2D + DirectWrite through the windows crate (no extra dependencies): rounded corners, alpha, crisp text, per-monitor DPI. Layered topmost popup window (WS_EX_TOPMOST | WS_EX_TOOLWINDOW).
  • Both dialogs share one window class and renderer; layout differs (horizontal icon row vs. vertical list).
  • Theme: auto reads AppsUseLightTheme from the registry; colors are two small palettes.

Tray & single instance

  • Shell_NotifyIcon; icon is a “§” glyph embedded as .ico resource. Right-click → menu → Quit.
  • Named mutex Global\win-app-switcher-single-instance. If already held: MessageBox Yes/No → on Yes, broadcast a registered window message telling the old instance to exit, wait for the mutex, continue; on No, exit.

5. Configuration

config.toml lives next to the exe. Missing file or missing keys → defaults below. config.example.toml ships with every line commented out:

# win-app-switcher configuration.
# Copy to config.toml next to the exe and uncomment what you want to change.
# Missing keys fall back to the defaults shown here.

# Dialog theme: "auto" follows Windows light/dark mode. ("auto" | "light" | "dark")
#theme = "auto"

# Dialog size multiplier.
#scale = 1.0

# Show the selected application's name under its icon.
#show_selected_name = true

# Which windows to list: only the current virtual desktop, or all. ("current" | "all")
#desktop_filter = "current"

# Restore minimized windows when activated.
#restore_minimized = true

# Which monitor the dialog opens on. ("mouse" | "primary")
#dialog_monitor = "mouse"

# WIN+§ shows the window list only after this delay (ms).
# A quicker tap switches windows with no UI.
#dialog_delay_ms = 150

6. Crates & repo layout

  • windows — all Win32/COM APIs (hook, D2D, DWrite, shell, tray, virtual desktops).
  • serde + toml — config.
  • anyhow — error plumbing at startup; hot paths don’t allocate errors.
  • Nothing else. No GUI framework, no async runtime.
src/
  main.rs      // startup, single instance, tray, main message loop
  hook.rs      // WH_KEYBOARD_LL state machine, WIN suppression
  apps.rs      // enumeration, eligibility, grouping, icons, activation
  ui.rs        // both dialogs: layout + Direct2D rendering, mouse/key input
  config.rs    // TOML load + defaults

8. Risks

RiskMitigation
Start-menu flash on WIN releaseDummy-key injection before passing WIN up; proven by AAS and PowerToys.
Stuck modifier keys after switchPost-session GetAsyncKeyState sweep + corrective SendInput (port of AAS RestoreKey).
SetForegroundWindow refusalsAttachThreadInput dance + AllowSetForegroundWindow; hung-window timeout guard.
Runs unelevated (user skips admin)Priority silently degrades to HIGH and elevated apps become invisible to the hook while focused. Detect at startup (TokenElevation) and warn once via a balloon notification; document the Task Scheduler setup as the fix.
REALTIME starves the systemOnly possible with a runaway loop — all threads block on GetMessage/events by design. Code review gate: no polling loops.
UWP icon edge casesshell:AppsFolder path is the OS’s own resolver; fall back to a generic icon.
